How would you implement Google Maps?

use postgresql(postgis extension) as databases

MongoDB 原生支持地理位置索引，可以直接用于位置距离计算和查询。

Naively, we could represent the world as a graph, using locations as vertices, roads as the edges, and weighting each edge by the average time it takes to traverse it. Then we could use an algorithm like Dijkstra’s or A* to find the shortest path between two points.

However, the number of locations is huge, and so is the number of map queries, so the naive strategy would produce unacceptably slow performance. I would optimize it by

  • Dividing the graph into different “zoom levels”. Most roads lie within cities, and only a few roads go between cities. So if we used cities as vertices instead of buildings, we could perform computations on a much smaller graph.

If I wanted to drive from Microsoft (in Redmond, WA) to Google (in Mountain View, CA), first I would look at the state graph and find a path from Washington to California. Then I would look at the city graph and find a path from Northern California to Mountain View. Then I would look at the local graph (which would only include vertices in Mountain View), and return the union of these three paths. This wouldn’t return the absolute shortest path, but it would be a good approximation and something that people could use.

  • Caching shortest paths. This would save a lot of time, since a lot of users type the same queries as other users, and we don’t want to do extra computation where we don’t have to.

Caching data is a standard trick in computer science, but it gets complicated because Google does traffic prediction and bases its shortest paths on real-time information. Every time the information changes, Google would have to recompute all the shortest distances, which would take unacceptably long.

The naive way to deal with this would be by removing all possible combinations of k edges from the local graph, and caching shortest paths for each combination. Then we compute the real-time travel times for each cached path (summing the travel times for each edge along the path), and return the one that happens to be shortest at that particular moment. However, this would require us to solve the

multiple-fault replacement paths problem, which has no known practical solution (a nice attempt is here).

So we should look for approximate solutions. For each road, find the mean and standard deviation of the time it takes to travel that road. Then regenerate the local graph k times, where the edge weights are sampled according to a Gaussian distribution with those parameters. This produces k shortest path candidates, which we can compare to each other in real time.
